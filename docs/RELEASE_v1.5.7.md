# Release v1.5.7 — Фикс «фантомной» ошибки при `/result` и других команд бота

## Дата
Май 2026

## Краткое описание
Расширение фикса из v1.5.6 на остальные хендлеры Telegram-бота. После v1.5.6 регистрация турнира по фото снова работала, но при добавлении результата командой `/result` для турнира `MAIN EVENT Day 1_A` пользователь видел:

```
❌ Ошибка при добавлении результата. Попробуйте еще раз.
```

При этом фактически результат успешно сохранялся в БД (повторный `/result` возвращал «У вас нет турниров без результатов»). Симптом тот же, что в v1.5.6, но точка падения — success-сообщение в `handleResultInput`, а не в `photoHandler`.

## Корневая причина
Та же, что в v1.5.6 — Telegram отвергает сообщение с `parse_mode: "Markdown"`, если в тексте есть незакрытый спец-символ (`_`, `*`, `` ` ``, `[`). v1.5.6 покрыл только `src/bot/handlers/photoHandler.ts`, но в `src/bot/commands.ts` оставалось ещё 10 мест, где имя турнира/площадка/имя пользователя/строка ошибки подставлялись в Markdown-шаблон без экранирования:

1. `link()` — `result.user.username`, `result.error`.
2. `handleTournamentRegistration()` success — `newTournament.name`, `newTournament.venue`.
3. `handleResultInput()` success — `updatedTournament.name` (THE bug).
4. `getStats()` — `currentVenue` в двух местах (error и заголовок).
5. `listTournaments()` — `tournament.name`, `tournament.venue` в цикле.
6. `finalizeTournamentEdit()` success — `tournament.name`, `tournament.venue`.
7. `showCurrentVenue()` — `currentVenue`.
8. `setCurrentVenue()` — `venueName` (пользовательский ввод).

Технически путь падения в `handleResultInput`:
1. API PUT `/api/tournaments/[id]` отрабатывает успешно, БД пишет результат.
2. Бот пытается отправить «✅ Результат добавлен! 🎰 **${updatedTournament.name}**…» с `parse_mode: "Markdown"`.
3. Telegram возвращает `400: Bad Request: can't parse entities` из-за `_` в имени.
4. `ctx.reply` бросает исключение → срабатывает внешний `catch` → `await ctx.reply("❌ Ошибка при добавлении результата. Попробуйте еще раз.");`.
5. Пользователь думает, что результат не сохранился, хотя в БД он уже есть.

## Что исправлено
В `src/bot/commands.ts` все 10 мест с подстановкой OCR/БД/error-данных в Markdown-сообщения теперь оборачивают значение в `escapeMarkdown(...)` (из v1.5.6). Никакой новой логики, никаких новых паттернов — только применение уже существующей утилиты.

Текст в `inline_keyboard` button (`text: \`🎰 ${tournament.name} ($${tournament.buyin})\``) сознательно НЕ экранируется: текст кнопок в Telegram Bot API — plain text, `parse_mode` на них не действует.

`src/bot/services/notificationService.ts` сейчас используется только в тестах (в проде не зовётся) — фикс там не делал, чтобы не вносить лишних изменений; когда сервис будет включён в прод, экранирование надо будет добавить и туда (или применять `escapeMarkdown` на стороне вызывающего кода перед `ctx.reply`).

## Файлы
- `src/bot/commands.ts` — импорт `escapeMarkdown`, экранирование в 10 точках.
- `src/__tests__/bot/commands.test.ts` — добавлен интеграционный регресс-тест `handleResultInput`: при имени турнира `MAIN EVENT Day 1_A` пользователь должен получить success-сообщение с экранированным именем, а не fallback «Ошибка при добавлении результата».

## Тесты
- `npm test` — **475/475** зелёные (+1 регресс-тест к v1.5.6).
- `npm run lint` — 0 ошибок.

## Регресс-сценарий
1. Создать через бота турнир `MAIN EVENT Day 1/A` по фото билета (v1.5.6 уже починила сам этот шаг).
2. Дождаться окончания турнира, отправить `/result`.
3. Выбрать турнир в списке, ввести любой результат (например, `0`).
4. **До фикса:** «❌ Ошибка при добавлении результата. Попробуйте еще раз.», но в БД результат уже сохранён.
5. **После фикса:** «✅ Результат добавлен! 🎰 **MAIN EVENT Day 1\_A** …».

То же относится к `/tournaments`, `/stats`, `/venue`, `/setvenue` и `/link` — раньше любая из этих команд могла упасть, если в БД есть имя/площадка/username с `_`, `*`, `` ` `` или `[`.

## Деплой
Релиз катится через стандартный Vercel-флоу: после пуша в `main` Vercel автоматически собирает и деплоит. Бот в режиме `polling` подхватит новую версию после рестарта инстанса; в режиме `webhook` — мгновенно после деплоя.

После деплоя проверить:
- `/api/debug/version` возвращает `1.5.7`.
- В Vercel Logs не должно быть `400: Bad Request: can't parse entities` ни от одной команды бота.
